META DATA USE STATEMENT FOR META oEMBED READ
============================================

Application: 3D Hiroba
App ID: 26224957440539490
Purpose: Display official Instagram embeds for curated community posts.

---

OVERVIEW
--------
3D Hiroba is a community gallery platform for sharing 3D pen creations.
Administrators curate publicly available Instagram posts as featured items.
For each featured post we call the Meta oEmbed Read endpoint to obtain the
official embed HTML, which is rendered inside a modal dialog exactly as
Instagram provides it. The gallery grid itself shows only a small thumbnail
derived from the public post page's og:image meta tag; this thumbnail is
temporarily cached, refreshed every 7 days, and deleted on removal.

---

SUBMISSION FORM USE CASE DESCRIPTION (verbatim)
-----------------------------------------------

3D Hiroba is a community gallery for children sharing 3D pen creations.
Administrators curate publicly available Instagram posts as featured items.
For each featured post, we call the Meta oEmbed Read endpoint to obtain the
official embed HTML, which is rendered inside a modal dialog exactly as
Instagram provides it (using hidecaption=true and omitscript=true, with
embed.js loaded from www.instagram.com). The gallery grid itself does not
reproduce Instagram content; it shows only a small thumbnail obtained from
the publicly accessible post page — first the post page's og:image meta
tag, falling back to the publicly embeddable view at
www.instagram.com/p/<shortcode>/embed/captioned/ (the same view that
embed.js renders inside the iframe). This thumbnail is temporarily cached
in our CDN-backed object storage to reduce load on Instagram's image
servers, is automatically refreshed every 7 days by a scheduled job, and
is immediately deleted when the administrator removes the post. We do not
store engagement metrics, author profile data, or any non-public
information. We do not combine Instagram data with other datasets, do not
use it for advertising, and do not share it with third parties.

---

DATA COLLECTED
--------------

From the Meta oEmbed Read endpoint
(graph.facebook.com/v19.0/instagram_oembed):
- html  - the official Instagram embed HTML used inside our modal.

Parameters sent on every request:
- url=<canonical Instagram permalink>
- access_token=<APP_ID>|<APP_SECRET>
- omitscript=true     (script loaded once from www.instagram.com)
- hidecaption=true
- fields=html

We do NOT request, parse, or persist any of the following:
- author_name / author_url
- thumbnail_url / thumbnail_width / thumbnail_height
- title
- engagement counts (likes, comments)
- private profile or follower data

Gallery thumbnail
- A small image obtained by parsing the og:image meta tag from the public
  Instagram post page (the same image any web browser or link preview
  generator receives).
- Stored temporarily in Vercel Blob at picks/<shortcode>.<ext>.

---

DATA USE & PURPOSE
------------------

PRIMARY USE: Render the official Instagram embed for each featured post.
- The detail modal contains the unmodified embed HTML returned by oEmbed
  Read, processed by the official embed.js from www.instagram.com.
- The gallery list shows only a small cached og:image thumbnail with the
  post title; clicking the card opens the official embed.

NOT USED FOR:
- Targeted advertising.
- User profiling or tracking.
- Selling or sharing with third parties.
- Training machine learning models.
- Re-identification of users.
- Combining Instagram data with other datasets.

---

DATA STORAGE & SECURITY
-----------------------

curation.json on Vercel Blob (admin curation metadata)
- pick id, internal title, administrator-entered comment, the canonical
  permalink, the cached og:image Blob URL, the embed HTML returned by
  oEmbed Read, and ogImageRefreshedAt timestamp.
- Written only by the admin API; read by the public site.

Vercel Blob (picks/<shortcode>.<ext>)
- Temporary cache of og:image; overwritten in place every 7 days by a
  Vercel Cron job, and deleted immediately when the pick is removed.

No other storage location is used. Firestore, third-party CDNs, or
analytics platforms are not involved.

Security
- HTTPS only.
- Admin actions gated by httpOnly + Secure auth cookie.
- Server-side validation: shortcode regex extraction, canonical permalink
  normalization (https://www.instagram.com/p/<shortcode>/).

---

DATA RETENTION & DELETION
--------------------------

Retention
- Curation metadata is kept while the post is featured.
- Cached og:image and oEmbed HTML are refreshed every 7 days; we treat
  them as a transient cache, not as a persistent copy of Platform Data.
- Removing a pick from the admin dashboard immediately removes both the
  curation entry and the cached Blob (mediaUrl and thumbnailUrl).

User Rights
- Anyone can request deletion via help@scrib3dpen.jp or /contact.
- We honor takedown requests within 24 hours of confirmation.

Refresh Job
- Path: /api/cron/refresh-thumbnails
- Schedule: Sunday 03:00 UTC (vercel.json cron entry).
- Authentication: Authorization: Bearer <CRON_SECRET>.
- Behavior: for each pick older than 7 days, refetch og:image, overwrite
  the existing Blob, update ogImageRefreshedAt.

---

THIRD-PARTY SHARING
--------------------

No third party receives Instagram-derived data from us.
- Vercel acts as data processor (Blob storage, hosting).
- Meta receives the post URL and our app token when we call oEmbed Read.

API USAGE PATTERN
-----------------

1. Admin pastes an Instagram URL into the management interface.
2. Server normalizes it to https://www.instagram.com/p/<shortcode>/.
3. Server calls Meta oEmbed Read once and stores html.
4. Server fetches the public page once and extracts og:image, downloads
   the image, stores it at picks/<shortcode>.<ext> with allowOverwrite.
5. The pick is shown in the gallery; clicking opens the modal with the
   official embed.
6. Weekly cron refreshes og:image only; oEmbed HTML is not re-fetched.
7. Pick deletion removes the curation entry and the cached Blob.
